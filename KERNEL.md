# KERNEL.md — Chat Kernel Architecture

> Design doc for Alpha-App's multi-chat kernel layer.
> Written Tue Mar 3 2026. Status: APPROVED — ready for Phase 1.

## Overview

Today Alpha-App runs a single chat. One `MannekinClient` singleton wraps one
`AlphaClient` wrapping one `claude` subprocess. Switch sessions? Stop
everything, recreate. It works, but it's a mannequin with one dress.

The kernel architecture replaces this with **Chat objects** — each chat owns
its own subprocess, manages its own lifecycle, and knows how to sleep and
wake up. Multiple chats coexist. The WebSocket multiplexes between them.
The sidebar shows them with indicator lights.

This is not "build a multi-tab chat app." This is a foundation for:
- Multiple concurrent conversations (eventually with different models/souls)
- Instant startup (one in the chamber — subprocess pre-warmed before needed)
- Graceful lifecycle (idle reaping, resurrection on return)
- Clean identity (our nanoid chat IDs, Claude's UUIDs hidden away)

## The Chat Object

The **Chat** is the first-class entity. Everything else is something it owns.

```python
class Chat:
    # Identity
    id: str                        # nanoid (12 chars), generated by backend
    session_uuid: str | None       # Claude's session ID, None until first ResultEvent

    # State
    state: ChatState               # STARTING | IDLE | BUSY | DEAD
    title: str                     # First user message, truncated to 80 chars
    created_at: float              # Unix timestamp
    updated_at: float              # Last activity (refreshed on every send)

    # Subprocess (private)
    _client: AlphaClient | None    # Today: mannequin. Tomorrow: AlphaAI.
    _reap_task: asyncio.Task | None  # Idle timeout watchdog

    # Lifecycle
    async def send(content: list[dict]) -> None       # IDLE→BUSY
    async def events() -> AsyncIterator[Event]        # Stream from subprocess
    async def interrupt() -> None                     # Cancel current turn
    async def reap() -> None                          # Kill subprocess, go DEAD
    async def resurrect(session_uuid: str) -> None    # DEAD→STARTING (--resume)

    # Persistence
    def serialize() -> dict                           # For Redis
    @classmethod
    def deserialize(data: dict) -> Chat               # From Redis

    # Construction
    @classmethod
    def from_holster(cls, id: str, client: AlphaClient) -> Chat:
        """Create a Chat with a pre-warmed client. Starts in IDLE."""
        ...
```

What the Chat **wraps** is intentionally abstract. Today it creates an
`AlphaClient(model=HAIKU, system_prompt="")` — the mannequin, no soul.
Tomorrow it creates an `AlphaAI` with the full identity. The Chat manages
lifecycle, not identity. The dress changes; the hanger is the same.

New chats receive their subprocess from the **Holster** (see below) and
are born IDLE — warm and ready. The Chat never has to spawn its own
subprocess for the first conversation, only for resurrection.

### The "manager" is just a dict

```python
# In main.py lifespan or a thin module
chats: dict[str, Chat] = {}
```

No `KernelManager` class. No boss. The dict is a phone book. Convenience
functions (create, list, get, reap_all) can live as module-level functions
or a thin wrapper, but the Chat is the unit of work.

## The Holster

One in the chamber. The app always keeps a single warm `claude` subprocess
ready for the next new chat. When it gets used, another one starts warming
in the background.

```python
class Holster:
    """Always keeps one warm claude ready for the next new chat."""
    _warm: AlphaClient | None = None
    _warming: asyncio.Task | None = None

    async def warm(self) -> None:
        """Start warming a new client in the background."""
        self._warming = asyncio.create_task(self._do_warm())

    async def _do_warm(self) -> None:
        """Spawn a claude subprocess (new session) and wait until it's ready."""
        client = AlphaClient(
            model=MODEL,
            system_prompt="",
            permission_mode="bypassPermissions",
        )
        await client.start(None)    # New session, no --resume
        self._warm = client
        self._warming = None

    async def claim(self) -> AlphaClient:
        """Take the warm client. Immediately start warming a replacement."""
        if self._warm is not None:
            client = self._warm
            self._warm = None
            await self.warm()       # Chamber the next round
            return client

        # Rare: holster empty (replacement still warming or nothing started)
        if self._warming is not None:
            await self._warming     # Wait for it
            return await self.claim()

        # Nothing at all — cold start
        await self._do_warm()
        return await self.claim()

    async def shutdown(self) -> None:
        """Clean up on app shutdown."""
        if self._warming:
            self._warming.cancel()
        if self._warm:
            await self._warm.stop()
```

### How it works

1. **App starts up** → `holster.warm()` — one `claude` starts in the background
2. **User clicks "New Chat"** → `holster.claim()` — grab the warm subprocess,
   wrap it in a Chat (born IDLE), immediately start warming a replacement
3. **User clicks "New Chat" again** → `holster.claim()` — if the replacement
   is ready, instant. If still warming, wait (usually <2 seconds, and the
   user is navigating/clicking during that time anyway)
4. **Nobody clicks anything for an hour** → one warm subprocess sits idle.
   That's fine. It's one process. (Note: the holster subprocess is NOT on the
   reap timer — only Chats get reaped. The holster stays warm indefinitely.)

### Why this is better than warm-on-create

- **Zero startup delay for new chats.** The subprocess is warm before the
  user even thought about starting a conversation.
- **Ten browser tabs don't create ten subprocesses.** The holster is
  server-side. One warm `claude`. Period.
- **The app is ready before the user is.** Open browser, start typing.
  The subprocess was warm before Chrome finished loading.
- **Clean separation.** The Holster owns "keeping a spare ready." The Chat
  owns "managing an active conversation." Neither does the other's job.

## State Machine

```
  from holster (new chat)                      ┌──────┐
  ─────────────────────────────────────────── > │ IDLE │
                                                └──┬───┘
                                                   │
                                         send()    │
                                                   v
              reap timer / fatal error         ┌──────┐
        ┌──────────────────────────────────── │ BUSY │
        │                                      └──┬───┘
        │                                         │
        v           turn complete (ResultEvent)    │
   ┌──────┐         or interrupt                   │
   │ DEAD │ <─────────────────────────────────────┘
   └──────┘
        │
        │  send() (resurrection via --resume)
        v
   ┌──────────┐    subprocess ready    ┌──────┐
   │ STARTING │ ────────────────────> │ IDLE │
   └──────────┘                        └──────┘
        │
        │ startup failure
        v
   ┌──────┐
   │ DEAD │
   └──────┘
```

### States

| State | Subprocess | What's happening |
|-------|-----------|------------------|
| **IDLE** | Running | Subprocess alive, waiting for a message. Reap timer ticking. |
| **BUSY** | Running | Actively processing a turn. Streaming events back. |
| **DEAD** | None | Reaped, errored, or killed. Metadata survives in Redis. |
| **STARTING** | Spawning | Resurrection only. `claude` subprocess coming up via `--resume`. 1-2 sec. |

### Transitions

| From | To | Trigger |
|------|-----|---------|
| *(new from holster)* | **IDLE** | `create-chat` — claim pre-warmed subprocess, born ready |
| IDLE | BUSY | `send()` (reap timer resets) |
| BUSY | IDLE | Turn complete (`ResultEvent`). Reap timer starts. |
| BUSY | IDLE | Interrupt (turn cancelled, subprocess lives) |
| BUSY | DEAD | Fatal error, subprocess crash |
| IDLE | DEAD | Reap timer fires (10 min default) |
| DEAD | STARTING | `send()` to a dead chat (resurrection via `--resume`) |
| STARTING | IDLE | Subprocess ready, resume events drained |
| STARTING | DEAD | Startup failure |

### Key insight

New chats **skip STARTING entirely**. They're born IDLE because the Holster
already did the waiting. STARTING only exists for resurrection — bringing a
DEAD chat back to life with `--resume`. This is a less common path (the user
returning to an old, reaped chat) and a 1-2 second delay there is acceptable.

### Rules

1. **Multiple chats can be BUSY simultaneously.** Each has its own subprocess.
   The backend doesn't serialize turns across chats.
2. **The reap timer starts when IDLE begins and resets on every `send()`.**
   Asyncio watchdog pattern:
   ```python
   def _reset_reap_timer(self):
       if self._reap_task:
           self._reap_task.cancel()
       self._reap_task = asyncio.create_task(self._reap_after(600))
   ```
3. **Resurrection uses `--resume`.** When a DEAD chat gets a new `send()`,
   the Chat spawns a fresh subprocess with the stored session UUID. Claude
   picks up where it left off. Resume event draining happens in STARTING.
4. **Interrupt kills the turn, not the subprocess.** BUSY → IDLE.
5. **The holster subprocess is not on the reap timer.** It stays warm
   indefinitely until claimed. Only Chats get reaped.
6. **At most one warm-but-unused Chat at a time.** If the user creates a
   new chat (IDLE, no messages) and then tries to create another, the
   "New Chat" button is grayed out or requires confirmation. This prevents
   orphaned warm subprocesses from accumulating. (The holster itself is
   always exactly one subprocess — the guard is about unclaimed Chats.)

## Identity

Two ID spaces, one mapping:

| ID | Format | Source | Visible to user |
|----|--------|--------|-----------------|
| **Chat ID** | nanoid, 12 chars | **Backend** generates on `create-chat` | Yes: URL bar, sidebar |
| **Session UUID** | Claude's format | Claude, via first `ResultEvent` | Discreet: popover with copy |

**Why backend-generated?** The backend is the source of truth for chat
identity. It creates the Chat, generates the nanoid, and returns it to the
frontend. The round-trip is negligible (localhost or LAN) and buys us
authority — no ambiguity about who owns identity.

**Why the UUID stays accessible?** Sometimes you need to dig into a `claude`
JSONL transcript directly. The UI has a discreet chat info popover showing:
- Chat ID: `V1StGXR8_Z5j` \[copy\]
- Session: `a1b2c3d4-e5f6-...` \[copy\] *(only after first exchange)*

Accessible but not in the way.

### nanoid spec

- **Alphabet:** Default nanoid (A-Za-z0-9_-). URL-safe, no encoding needed.
- **Length:** 12 characters. ~71 bits of entropy. Collision probability at
  1000 chats: effectively zero.
- **Libraries:** `nanoid` (npm) for frontend, `nanoid` (PyPI) for backend.
  Same algorithm, same output space.

## Serialization (Redis)

### Per-chat metadata

```
Key:    alpha:chat:{chatId}
Type:   Hash
TTL:    29 days (refreshed on activity)

Fields:
  session_uuid   — Claude's session ID (empty until first ResultEvent)
  title          — First user message, truncated to 80 chars
  state          — Last known state before server shutdown (for restart recovery)
  created_at     — Unix timestamp
  updated_at     — Unix timestamp (refreshed on every send)
```

### Chat index (for sidebar listing)

```
Key:    alpha:chats
Type:   Sorted Set
Score:  updated_at (float, Unix timestamp)
Member: chatId
```

### What persists vs. what doesn't

| Persists (Redis) | Does NOT persist |
|-------------------|-----------------|
| Chat ID | The subprocess itself |
| Session UUID | In-flight streaming state |
| Title, timestamps | The reap timer |
| Last known state | Message content (that's in JSONL) |

**On server restart:** All chats come back as DEAD. Subprocesses are gone.
The holster warms a fresh `claude` during startup. Next `send()` to any
existing chat triggers resurrection via `--resume`. The sidebar shows old
chats as DEAD (gray dot) until the user interacts with them.

### Why 29 days?

`claude` auto-prunes session transcripts at approximately 30 days. By
expiring our Redis metadata at 29 days, the reference dies before the thing
it references. The sidebar entry disappears slightly before the JSONL
transcript would be pruned, so we never have a chat pointing to a deleted
transcript.

In the brief window between day 29 (Redis gone) and day ~30 (JSONL pruned),
the transcript still exists on disk but isn't listed. Nobody notices. Nothing
breaks. If someone truly needs an old transcript, they can find the JSONL by
session UUID through other means — but the sidebar stays clean.

## WebSocket Protocol v2

One WebSocket, all chats multiplexed. The chatId field is on everything.

### Client → Server

```jsonc
// Create a new chat (backend claims from holster, returns chatId)
{ "type": "create-chat" }

// Send a message to a specific chat
{
  "type": "send",
  "chatId": "V1StGXR8_Z5j",
  "content": "Hello"                    // or content block array
}

// Interrupt a specific chat's current turn
{ "type": "interrupt", "chatId": "V1StGXR8_Z5j" }

// Request chat list (sidebar hydration on connect)
{ "type": "list-chats" }
```

### Server → Client

```jsonc
// Chat lifecycle
{ "type": "chat-created", "chatId": "V1StGXR8_Z5j", "data": { "state": "idle" } }
{ "type": "chat-state",   "chatId": "V1StGXR8_Z5j", "data": { "state": "busy" } }

// Streaming events — same as v1 but with chatId
{ "type": "text-delta",     "chatId": "...", "data": "chunk" }
{ "type": "thinking-delta", "chatId": "...", "data": "chunk" }
{ "type": "tool-call",      "chatId": "...", "data": { "toolCallId": "...", "toolName": "...", "args": {}, "argsText": "..." } }
{ "type": "tool-result",    "chatId": "...", "data": { "toolCallId": "...", "result": ..., "isError": false } }

// Turn lifecycle
{ "type": "done",        "chatId": "..." }
{ "type": "interrupted", "chatId": "..." }
{ "type": "error",       "chatId": "...", "data": "something broke" }

// Chat list response (sidebar hydration)
{
  "type": "chat-list",
  "data": [
    { "chatId": "...", "title": "...", "state": "idle", "updatedAt": 1709500000 },
    { "chatId": "...", "title": "...", "state": "dead", "updatedAt": 1709490000 }
  ]
}
```

### Breaking change

Yes. This is a clean break from v1. The `sessionId` field is gone. All
messages carry `chatId`. The backend no longer emits `session-id` events —
it emits `chat-created` with the chatId.

We don't need backward compatibility. There's one user and we control
both ends.

### The create-chat round-trip

When the user clicks "New Chat":
1. Frontend sends `{ "type": "create-chat" }`
2. Backend calls `holster.claim()` — instant if warm, brief wait if not
3. Backend generates nanoid, creates Chat (born IDLE), persists to Redis
4. Backend sends `{ "type": "chat-created", "chatId": "...", "data": { "state": "idle" } }`
5. Frontend navigates to `/chat/{chatId}`, user starts typing
6. User hits send — message goes to the already-warm subprocess. Zero delay.

If the holster is still warming (rare — only if two chats are created in
rapid succession), step 2 takes 1-2 seconds. The frontend can show a brief
spinner or just let the user type in the old chat until the new one is ready.

## Frontend Contract

### Store (Zustand)

The store becomes multi-chat aware:

```typescript
interface ChatMeta {
  id: string;               // nanoid
  title: string;
  state: ChatState;         // "starting" | "idle" | "busy" | "dead"
  updatedAt: number;
}

interface WorkshopState {
  // Multi-chat
  chats: Record<string, ChatMeta>;
  activeChatId: string | null;

  // Per-chat messages
  messages: Message[];               // Active chat's messages
  messageCache: Record<string, Message[]>;  // Recently viewed chats' messages

  // Global
  connected: boolean;
  contextPercent: number;
  model: string;
  // ... existing fields, minus isRunning (now per-chat via ChatMeta.state)
}
```

`isRunning` goes away as a global — it's derivable from
`chats[activeChatId]?.state === "busy"`.

### Message caching

On chat switch, messages are loaded from the backend (JSONL via HTTP). The
active chat's messages plus recently viewed chats are cached in
`messageCache` to avoid re-fetching on quick back-and-forth switches.

**Streaming into non-active chats:** When a background chat is BUSY, incoming
deltas update its `messageCache` entry. Switching back to that chat shows
the latest state immediately — no re-fetch, no lost streaming. The sidebar
indicator light updates in real-time regardless of which chat is active.

Performance needs benchmarking with large chats (thousands of messages).
If JSONL loading is <200ms, lazy loading is fine. If it's >1s, caching
becomes more important.

### Sidebar

- Lists chats from `chats` map, sorted by `updatedAt` descending
- Each entry: title (truncated), indicator light, relative timestamp
- **Indicator lights:**
  - 🟢 Green: IDLE (alive, waiting)
  - 🟡 Amber pulsing: BUSY or STARTING (working)
  - 🔴 Red: Error (subprocess crashed)
  - ⚫ Gray: DEAD (no subprocess) — or no dot at all if gray dots are ugly
- "New Chat" button sends `create-chat` over WebSocket
  - **While an unused warm chat exists (IDLE, no messages):** Button is
    grayed out or enters a confirmation state (click once = highlight,
    click again = confirm and replace). Prevents orphan accumulation.
- Click a chat: set `activeChatId`, load messages from cache or backend
- The current (warm, no messages) chat appears in the sidebar as the active
  entry — "New chat" in italics, green dot.

### Chat ID Popover

Somewhere discreet on the active chat (title area? a ⓘ icon?):
- **Chat ID:** `V1StGXR8_Z5j` \[copy icon\]
- **Session:** `a1b2c3d4-e5f6-7890-...` \[copy icon\]
- Session line only appears after first exchange (when UUID is known)

### Routing

```
/chat              → New chat (sends create-chat, redirects to /chat/:chatId)
/chat/:chatId      → Specific chat (load from backend if exists, resurrect if DEAD)
```

Chat IDs in URLs are nanoids, not UUIDs.

## Migration Plan

### Phase 1: Chat + Holster (backend refactor, no visible change)

Build `chat.py` with Chat class, Holster class, and ChatState enum.

- Chat wraps what MannekinClient does today: create AlphaClient, send
  messages, stream events, track session UUID. Add state machine and
  reap timer.
- Holster keeps one warm `claude` ready. `claim()` returns the pre-warmed
  client and starts warming a replacement.
- Replace the global `client` singleton in `ws.py` with a Holster +
  single Chat. Same WebSocket protocol (no chatId yet). Same behavior.
  Different internals.

**Ship when:** Existing app works identically. The refactor is invisible.

### Phase 2: Multi-chat (backend + frontend, protocol break)

The big one. Backend and frontend ship together because the WebSocket
protocol changes.

Backend:
- Chat registry (dict) in app state
- `create-chat` handler: claim from holster, generate nanoid, return chatId
- ws.py routes messages by chatId
- New protocol messages (chat-created, chat-state, list-chats)
- Redis persistence for chat metadata (29-day TTL)
- Idle reaping with asyncio watchdog

Frontend:
- Store refactor (chats map, activeChatId, messageCache, per-chat state)
- Sidebar: chat list with indicator lights, "New Chat" with guard
- WebSocket: chatId on all messages, create-chat on "New Chat" click
- Routing: /chat/:chatId (nanoid)
- Message caching and streaming into non-active chats

**Ship when:** Multiple chats work. Create, switch, send, receive, reap.

### Phase 3: Polish

- Chat ID popover with copy buttons (nanoid + UUID)
- Indicator light animations (CSS pulse for amber)
- Chat title editing
- Delete chat (UI + Redis cleanup)
- Keyboard shortcuts (Cmd+N new chat, etc.)
- Empty state refinement
- Haiku title generation (nice-to-have)

## Decisions

Resolved during design review, Tue Mar 3 2026:

1. **Message history on chat switch:** Load from backend on switch. Cache
   active + recently viewed chats in `messageCache`. Streaming deltas for
   background BUSY chats update the cache so switching back is instant.
   Benchmark with large chats to validate — if loading is >1s, caching
   strategy needs expansion.

2. **Concurrent BUSY chats:** Supported. The backend runs separate
   subprocesses. The frontend caches messages for all active chats and
   updates sidebar indicators in real-time. Streaming deltas for non-active
   chats go into `messageCache` silently.

3. **Startup readiness:** The Holster solves it. App boots → holster warms
   a `claude` → user opens browser → clicks "New Chat" → instant. No warm-on-
   connect (wasteful with multiple tabs). No warm-on-click (adds delay).
   The app is ready before the user is.

4. **New Chat guard:** At most one warm-but-unused Chat at a time. "New Chat"
   button is grayed out or requires confirmation while an unused chat exists.
   The holster itself always holds exactly one warm subprocess — the guard is
   about unclaimed Chats, not the holster.

5. **Title generation:** First user message, truncated to 80 chars. Haiku
   title generation is Phase 3+ nice-to-have.

6. **Max concurrent subprocesses:** No hard limit. Idle reaping keeps it
   sane. Even at 2-4 GB per `claude`, Primer would need dozens to feel it.
   Not worth engineering around for a single-user app.

7. **29-day TTL:** `claude` prunes transcripts at ~30 days. Our Redis
   metadata expires at 29 days so the reference dies before the thing it
   references. No orphan sidebar entries.

8. **Chat switch animation:** Instant swap. Don't animate what doesn't need
   covering. If loading messages from the backend is <200ms, any crossfade
   or skeleton would be more disruptive than the raw pause. "Click-pause-
   react" beats "click-react-react." Revisit in Phase 3 only if measured
   latency warrants it.

9. **DEAD chat visual treatment:** Gray dot, normal text. The dot vocabulary:
   green = IDLE, amber = BUSY/STARTING, red = error, gray = DEAD. If a
   column of gray dots looks ugly in practice, fallback is: no dot for DEAD
   (presence of a dot = subprocess alive). Pixelfuck in Phase 3.

10. **Holster session UUID:** Captured on first `ResultEvent` after the first
    user message — not during warmup. Claude doesn't return a session ID
    until it responds to a message, so the holster's warm subprocess is
    sessionless until claimed and used. The Chat's `session_uuid` field
    starts as `None` and gets set when the first turn completes.
